# Plan: Fix Accessibility Issues in CalendarView

## Goal
Address the Web Interface Guidelines violations identified in the CalendarView component review, focusing on accessibility (ARIA labels, semantic elements, icon handling) and minor style consistency. All changes must preserve existing functionality, visual appearance, and behavior exactly.

## Scope
- **File to modify**: `src/components/CalendarView.tsx`
- **Related files** (read-only inspection only): `src/app/globals.css` (for any new class patterns), `src/components/Icons.tsx` (icon usage patterns)
- **Out of scope for this iteration** (to minimize risk):
  - Deep linking calendar view/date state to URL query params (would require router/search-params integration and potential hydration considerations; can be a follow-up).
  - Broader focus-visible ring styles or global CSS changes unless required for the flagged elements.
  - Any functionality changes to activity fetching, filtering, pagination logic, or react-big-calendar integration.

## Detailed Changes

### 1. Add aria-labels to icon-only navigation buttons (CustomToolbar)
- **Location**: Lines 88–96 (`nav-btn` buttons for PREV/NEXT) and the Today button context.
- **Action**:
  - Wrap ChevronLeft/ChevronRight icons with explicit `aria-label` on the `<button>` elements.
  - Suggested labels (context-aware):
    - PREV: `"Previous month"` / `"Previous week"` / `"Previous day"` depending on current `view`, or keep generic `"Go to previous period"`.
    - NEXT: `"Next month"` / `"Next week"` / `"Next day"`.
    - TODAY: Already has text label "Today" – ensure it remains the accessible name.
- **Why safe**: Purely additive ARIA attributes; no behavior change. Matches existing `onClick` handlers.

### 2. Add aria-labels and aria-disabled to agenda pagination buttons (CustomAgenda)
- **Location**: Lines 235–250 (ChevronLeft/ChevronRight pagination controls).
- **Action**:
  - Add `aria-label="Previous page"` and `aria-label="Next page"` to the respective buttons.
  - Change `disabled={...}` to also include `aria-disabled={currentPage === 0}` (and equivalent for next) for better screen-reader announcement.
  - Keep the existing `disabled` attribute for native behavior + opacity styling.
- **Why safe**: Enhances accessibility without altering pagination state logic or `goBack`/`goForward` callbacks.

### 3. Mark decorative Umbrella icon as hidden
- **Location**: Line 532 (inside `CalendarEventRenderer` for holiday events).
- **Action**: Add `aria-hidden="true"` to the `<Umbrella>` component instance (or the underlying SVG).
- **Why safe**: Icon is purely decorative alongside the event title text; does not convey additional meaning.

### 4. Improve accessibility of agenda table rows
- **Location**: Lines 293–321 (`<tr className="agenda-row clickable-row" onClick=...>`).
- **Options & recommendation** (user to confirm preference):
  - **Preferred (minimal layout impact)**: Keep `<tr>` but add:
    - `role="button"`
    - `tabIndex={0}`
    - `onKeyDown` handler that triggers `onSelectActivity` on Enter/Space (standard button key handling).
    - `aria-label` describing the row (e.g., `${activity.title} at ${time}`).
  - **Alternative (more semantic)**: Replace inner content with a `<button>` that spans the row using CSS (`display: contents` or table-cell buttons) while preserving current visual styling.
- **Rationale**: Guidelines prohibit bare click handlers on non-interactive elements. Table rows are common for "select item" patterns; adding role + keyboard support satisfies both accessibility and guidelines without major reflow.
- **Risk mitigation**: Test that row click, hover styles, and table layout remain identical.

### 5. Convert inline-styled action buttons to className (optional polish)
- **Location**: Lines 104–146 ("Own Responsibility" and "Create Activity" buttons inside CustomToolbar).
- **Action**:
  - Extract the repeated inline style objects into reusable CSS classes in `globals.css` (e.g., `.rbc-action-btn`, `.rbc-responsibility-btn`).
  - Replace `style={{ ... }}` with `className="rbc-action-btn"` (and variant for responsibility color).
- **Why safe & recommended**: Improves maintainability and consistency with other toolbar buttons (which already use classes like `rbc-create-btn`). Existing visual output (colors, padding, border-radius, etc.) will be replicated exactly in CSS.
- **Alternative**: Leave as-is if user prefers minimal diff.

### 6. Verification Steps (post-edit)
- Run the dev server and manually test:
  - All calendar navigation (month/week/day, prev/next/today).
  - Agenda pagination and row clicks (mouse + keyboard).
  - Holiday rendering (icon visibility).
  - Screen reader simulation (VoiceOver / browser dev tools) for labels and disabled states.
  - Confirm no visual regression in toolbar or agenda table.
- Run `npm run lint` and `npm run typecheck` (or equivalent) to ensure no new errors.
- No new tests required unless existing test suite covers CalendarView.

## Trade-offs & Open Questions for User

- **Agenda row approach**: Which of the two options above do you prefer for the `<tr>` fix? The role+keyboard approach keeps diff smaller.
- **Inline styles refactor**: Is extracting to CSS classes worth the extra change, or keep the current inline approach for these two buttons?
- Any other priority order or elements you want deprioritized?

## Execution Notes
- Make changes iteratively in small, reviewable edits.
- After implementation, re-run the web-design-guidelines review to confirm all items are resolved.
- Preserve all existing imports, hooks, and react-big-calendar props exactly.

This plan is ready for implementation once the above questions are clarified.

# Phase 2: Fix UI & Accessibility Issues in ResponsibilityForm (Own Responsibility Modal)

## Goal
Address the Web Interface Guidelines violations and structural issues identified in the ResponsibilityForm review for the "Own Responsibility" flow. All changes must preserve existing form submission, validation, recurrence logic, and visual appearance exactly.

## Scope
- **Primary file to modify**: `src/components/ResponsibilityForm.tsx`
- **Secondary file (read-only inspection)**: `src/app/page.tsx` (modal wrapper)
- **Related files** (read-only): `src/app/globals.css` (activity-form styles), `src/components/Icons.tsx`
- **Critical note**: The current code contains a call to undefined function `updateDurationFromTimes` (line 258). Any fix must restore functionality without side effects.
- **Out of scope**:
  - Changes to Responsibility creation API or data model.
  - Refactoring the entire form into a shared component with ActivityForm.
  - Adding new features (e.g., better date validation).

## Identified Issues (from review)

1. Misleading CSS class `activity-form` used for responsibility form.
2. Multiple decorative icons inside `<label>` elements missing `aria-hidden="true"`.
3. Duplicate "Category" form group (lines 239-248 and 268-277) — clear copy-paste error.
4. Checkbox for recurring event lacks proper label association.
5. Error and success banners lack `role` / `aria-live` for screen readers.
6. Submit button missing `aria-busy` during submission.
7. Cancel button icon missing `aria-hidden`.
8. Modal in page.tsx lacks `aria-labelledby` and focus management.
9. **Functionality bug**: `updateDurationFromTimes` is referenced but never defined.

## Detailed Changes

### 1. Fix critical missing function (restore functionality)
- **Location**: Add `updateDurationFromTimes` helper (or inline the logic where called).
- **Action**: Implement a small pure function that calculates duration from start/end and updates state safely.
- **Why required**: Current code would throw runtime error on End Date change. Must be fixed to keep "Own Responsibility" flow working.

### 2. Remove duplicate Category field
- **Location**: Lines 268-277 (second Category select).
- **Action**: Delete the duplicate block entirely.
- **Why safe**: First instance (lines 239-248) is correctly placed; duplicate serves no purpose and creates confusing UI.

### 3. Add `aria-hidden="true"` to all decorative icons in labels
- **Locations**: Lines 187, 210, 228, 239, 252, 268 (Tag, Calendar, Clock icons).
- **Action**: Add `aria-hidden="true"` to each icon component.
- **Why safe**: Purely presentational; improves accessibility without visual change.

### 4. Improve recurring checkbox accessibility
- **Location**: Lines 280-287.
- **Action**:
  - Wrap checkbox + label text in a proper `<label htmlFor>` or use `id` + `htmlFor`.
  - Add `aria-describedby` if needed for the recurrence options section.
- **Why safe**: Standard pattern; no behavior change.

### 5. Add live regions to error/success messages
- **Location**: Lines 312-313.
- **Action**:
  - Error: `role="alert"` + `aria-live="assertive"`
  - Success: `role="status"` + `aria-live="polite"`
- **Why safe**: Improves screen reader announcement during form submission flow.

### 6. Enhance submit button during loading
- **Location**: Line 317.
- **Action**: Add `aria-busy={isSubmitting}` and optionally `aria-disabled` when submitting.
- **Why safe**: Standard progressive enhancement.

### 7. Mark Cancel icon as decorative
- **Location**: Line 316.
- **Action**: Add `aria-hidden="true"` to `<X>` icon.
- **Why safe**: Icon is redundant with text "Cancel".

### 8. Rename CSS class for clarity (user decision: create new class)
- **Decision**: Create `responsibility-form` class (user preference).
- **Action**:
  - Change `className="activity-form"` → `className="responsibility-form"`.
  - In globals.css, add `.responsibility-form { ... }` that copies/extends the existing `.activity-form` styles to avoid visual regression.
- **Why safe**: Preserves exact look while improving semantics.

### 9. Modal accessibility improvements (page.tsx)
- **Location**: Around line 367 in page.tsx.
- **Action**:
  - Add `role="dialog"` and `aria-labelledby` to the modal container.
  - On open, auto-focus the Name input (user preference: first field).
- **Why safe**: Standard accessible modal pattern.
