# Plan: Simplify getSecurityContext to single userRole query

## Problem
`getSecurityContext` in `src/lib/auth.ts:5` performs two separate DB queries (`userRole` and `userGroupM2M`) and merges results in JS.  
`syncUserRoles` in `src/lib/prisma.ts:104` already writes the merged role set back to `userRole` after every group membership change.  
This causes:
- Redundant DB round-trip
- Inconsistent results in the window between a group change and sync completion

## Goal
Remove the `userGroupM2M` query from `getSecurityContext` and rely solely on the `userRole` table (kept in sync by `syncUserRoles`).

## Changes Required

### 1. `src/lib/auth.ts`
- Delete lines 11-22 (the entire `userGroupM2M` query block)
- Remove the JS merge logic on lines 26-28
- Keep the existing `userRole` query (lines 6-9) as the sole source of truth
- Simplify the return to just `Array.from(roles)` from the single query result

### 2. `src/lib/prisma.ts`
- No changes required
- `syncUserRoles` (lines 104-136) and its invocation in `userGroupM2M` hooks (lines 86-99) remain unchanged and continue to maintain the `userRole` table

## Verification Steps
1. Run existing tests (if any) that exercise role/group membership
2. Manually test: create group → assign role to group → add user to group → verify `getSecurityContext` returns the role without querying `userGroupM2M`
3. Confirm no regression in ACL enforcement that depends on `userContext.roles`

## Risks / Edge Cases
- Ensure `syncUserRoles` is always called on every `userGroupM2M` mutation path (already covered by the current hooks)
- Direct `userRole` inserts/deletes outside of the sync path must still be respected (they are preserved inside `syncUserRoles`)

- `src/lib/auth.ts` (only)
