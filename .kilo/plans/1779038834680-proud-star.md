# Fix AdminDashboard Pagination, CalendarView Fetching, and ActivityForm Normalization

## Overview
This plan addresses three performance and consistency issues:
1. Uniform server-side pagination pattern across all tabs in AdminDashboard.
2. Consolidate duplicate fetch effects in CalendarView.
3. Move leader/guide/observer normalization from UI component to API response layer.

All changes maintain existing behavior, use existing utilities (RECORDS_PER_PAGE, Pagination), and follow project patterns (useMemo, useCallback, secureFetch).

## 1. AdminDashboard.tsx – Uniform Pagination for All Tabs

### Current State
- RECORDS_PER_PAGE (10) and Pagination component exist.
- Only `participantsTab` applies `.slice(...)` on filtered results + renders `<Pagination>`.
- All other tabs (`rolesTab`, `groupsTab`, `groupMembersTab`, `groupRolesTab`, `userRolesTab`, `aclsTab`) perform client-side filtering but render the entire filtered array.
- `currentPage` is reset to 1 on every tab change (good).
- Filters are also reset on tab change.

### Changes
- Refactor each tab's useMemo to:
  - Compute filtered array (already done).
  - Compute `totalRecords = filtered.length`.
  - Slice the filtered array: `.slice((currentPage-1)*RECORDS_PER_PAGE, currentPage*RECORDS_PER_PAGE)`.
  - Append `<Pagination currentPage={currentPage} totalRecords={totalRecords} onPageChange={setCurrentPage} />` after the table (or inside the empty-state conditional appropriately).
- Extract a small helper (or inline) for the slice logic to avoid repetition.
- Ensure Pagination only renders when `totalRecords > RECORDS_PER_PAGE`.
- Update dependencies of each useMemo to include `currentPage` where necessary (but since page reset happens on tab/filter change, it should stay stable).
- No new state; reuse existing `currentPage`, `filters`, `setCurrentPage`.

### Files
- src/components/AdminDashboard.tsx (lines ~578–1165 for the tab memos)

### Verification
- Switch tabs with >10 filtered rows; confirm pagination appears and slicing works identically to Participants tab.
- Filter input resets page and shows correct total.

## 2. CalendarView.tsx – Consolidate Activity + Responsibility Fetches

### Current State
- Two separate `useEffect` blocks (lines 383–416 and 419–433), both keyed on `[refreshTrigger]`.
- Each performs its own fetch + state update → potential double re-render and race condition on rapid refreshTrigger changes.
- Responsibilities are merged later in `calendarActivities` memo.

### Changes
- Merge the two effects into a single `useEffect` that depends on `[refreshTrigger]`.
- Inside the effect, use `Promise.all([fetchActivities(), fetchResponsibilities()])` (or sequential awaits inside one async function).
- Keep the cancellation guard (`let cancelled = false`) and cleanup.
- Remove the second useEffect entirely.
- (Optional but recommended) keep the holiday fetch as a separate one-time effect.

### Files
- src/components/CalendarView.tsx (lines 383–433)

### Verification
- Trigger refresh (e.g. create activity); confirm single render cycle for activities + responsibilities.
- No duplicate network requests or stale state.

## 3. ActivityForm.tsx – Normalize at API Boundary

### Current State
- `formData` initialization (lines 272–274) defensively handles both plural (`leaders`) and singular (`leader`) shapes from `initialData`.
- This indicates possible inconsistency in API responses or legacy data.
- CalendarView also has similar defensive code (lines 401–403).

### Changes
- In the API layer (`/api/activities/route.ts` and `/api/activities/[id]/route.ts` if it returns single objects), ensure every activity object always exposes:
  - `leaders: string[]`
  - `guides: string[]`
  - `observers: string[]`
  (Already done for list endpoint; verify single-activity GET returns the same shape.)
- Remove the fallback expressions in ActivityForm.tsx:
  - Change `leader: (initialData as any)?.leaders || (initialData as any)?.leader || []` → `leader: initialData?.leaders ?? []`
  - Same for guide/observer.
- Optionally clean the similar defensive code in CalendarView (or leave it if it protects against other sources).
- Update TypeScript interfaces if needed so `ActivityData` and `ApiActivity` declare the arrays without optionals for legacy fields.

### Files
- src/app/api/activities/route.ts (already normalizes)
- src/app/api/activities/[id]/route.ts (verify)
- src/components/ActivityForm.tsx (lines 272–274)
- (Possibly) src/components/CalendarView.tsx (lines 401–403)

### Verification
- Edit an existing activity; confirm `initialData` always supplies arrays under the plural keys.
- No runtime fallback logic remains in the form.
