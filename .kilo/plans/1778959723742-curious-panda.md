# Revised Plan: Filter responsibilities by owner UUID in calendar

## Goal

Store `owner` as a user UUID for reliable identity matching, while keeping display name on the frontend.

## Problem with original approach

Matching by `owner` (display name string) is fragile вҖ” names can change or have duplicates. Storing the user's UUID (`ownerId`) makes the filter deterministic.

## Changes needed

### 1. Prisma schema (`prisma/schema.prisma`)

Add `ownerId String?` field to `Responsibility` model:

```prisma
model Responsibility {
  // ... existing fields ...
  owner    String?   // display name (kept for backward compat & UI)
  ownerId  String?   // user UUID (used for filtering)

  // ... system columns ...
}
```

Then run `npx prisma migrate dev --name add-owner-id` to generate the migration, and **then run `npx prisma generate`** to regenerate the Prisma client. The `$ npx prisma generate` step is critical вҖ” without it the client won't recognize the new `ownerId` field and will throw `Unknown argument \`ownerId\`. Did you mean \`owner\`?`.

### 2. Responsibility creation flow вҖ” propagate `ownerId`

**`src/app/page.tsx`** вҖ” `onOwnResponsibility()` (line 194):
- Add `ownerId: currentUser?.id || ''` to the new responsibility object passed to the form

**`src/components/ResponsibilityForm.tsx`**:
- Add `ownerId?: string` to the `ResponsibilityData` interface
- Add `ownerId: initialData?.ownerId ?? ''` to the `formData` state
- Use `UserTypeahead`'s `onSelect` callback (already exists, currently unused) to capture the selected user's `id`:
  ```tsx
  <UserTypeahead
    ...
    onSelect={(user) => setFormData(prev => ({ ...prev, ownerId: user.id }))}
  />
  ```
- Include `ownerId` in the POST payload

**`src/app/api/responsibilities/route.ts`**:
- Add `ownerId: z.string().optional().nullable()` to `responsibilitySchema`
- POST: store `ownerId` in the `data` object (alongside `owner`)

### 3. Responsibility read flow вҖ” filter by `ownerId`

**`src/app/api/responsibilities/route.ts`** вҖ” `GET` handler:
- Change the `where` clause to filter by `ownerId` instead of `owner`:
  ```typescript
  args: {
    where: {
      ownerId: securityContext?.id || undefined,
    },
    orderBy: { startDateTime: 'asc' }
  }
  ```
- `securityContext.id` is the user's UUID (already available, no changes to `auth.ts` needed)

### 4. Frontend вҖ” no display changes needed

- `ResponsibilityDetailModal.tsx` вҖ” already displays `responsibility.owner` (the name string), unchanged
- `CalendarView.tsx` вҖ” already reads whatever the API returns; `owner` name is passed through for display in the event/detail modal

### Files with no changes
- `src/lib/auth.ts` вҖ” `id` (UUID) is already in the security context
- `src/lib/prisma.ts` вҖ” no type changes needed (we use `securityContext.id` directly)
- `CalendarView.tsx` вҖ” already passes `r.owner` to calendar activities
- `UserTypeahead.tsx` вҖ” `onSelect` already provides the full `User` object with `id`

## Data flow summary

```
Create:
  Frontend: UserTypeahead.onSelect(user) вҶ’ formData.owner = user.name, formData.ownerId = user.id
  API POST:  stores owner (name string) + ownerId (UUID string)

Read:
  API GET:   WHERE ownerId = securityContext.id  (UUID match)
  Frontend:  renders r.owner (name string) for display
```

# Layout Fix: ResponsibilityForm uses undefined CSS classes

## Problem

`ResponsibilityForm.tsx` uses `form-grid` and `form-field` class names that have **zero CSS definitions** in any stylesheet. The fields stack without proper grid layout.

`ActivityForm` uses `form-row` (2-column CSS grid: `repeat(auto-fit, minmax(200px, 1fr))`) and `form-group` (flex column) вҖ” both properly defined in `globals.css`.

## Fix (single file: `src/components/ResponsibilityForm.tsx`)

1. **Replace `form-grid` вҶ’ `form-row`** вҖ” activates the existing 2-column grid
2. **Replace `form-field full-width` вҶ’ just `form-group`** inside a single-column `form-row` for full-width fields (Responsibility Name, Owner)
3. **Replace `form-field` вҶ’ `form-group`** for normal fields (Start, Duration, Category)
4. **Add `form-header`** with `<h2>Own Responsibility</h2>` matching `ActivityForm` style
5. **Remove the redundant editable End Date & Time field** вҖ” duration + start already compute end time; matches `ActivityForm` which only shows it as read-only
6. **Move recurring toggle into `form-group`** with the checkbox-label class (matching `ActivityForm`)
