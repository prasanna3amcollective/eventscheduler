# Plan: Fix RegisterForm onSuccess timing

## Problem
In `src/components/RegisterForm.tsx:165`, `onSuccess` is invoked inside `setTimeout(..., AUTO_ENROLL_DELAY_MS)` (2000 ms) **after** `await autoEnroll(...)`.
- When `pendingEventId` is present, `autoEnroll` performs a network request.
- On slow networks the 2-second timer can fire **before** the enroll request completes.
- Parent (`page.tsx:241`) receives the callback for a user that is not yet enrolled in the pending activity.

## Goal
Ensure `onSuccess` is only called **after** the auto-enrollment (if any) has finished, while preserving all current behavior for callers and UI flow.

## Analysis performed
- Read `src/components/RegisterForm.tsx` (full file).
- Located the single usage of `onSuccess` at `RegisterForm.tsx:164-166`.
- Confirmed `handleLoginSuccess` in `src/app/page.tsx:91-103` (the only consumer) simply sets user state and clears `pendingEventId`; it has no timing dependency on the 2-second delay.
- `AUTO_ENROLL_DELAY_MS` (line 57) is defined but used **only** for the problematic `setTimeout`.
- `autoEnroll` (lines 108-122) is already `async` and awaited (line 162), so the fix is straightforward.

## Proposed change (minimal, non-breaking)
**File:** `src/components/RegisterForm.tsx`

1. Delete the constant:
   ```ts
   const AUTO_ENROLL_DELAY_MS = 2000;   // remove entirely
   ```

2. Replace the delayed callback block (`RegisterForm.tsx:164-166`):
   ```ts
   if (onSuccess) {
     setTimeout(() => onSuccess(data as UserData), AUTO_ENROLL_DELAY_MS);
   }
   ```
   with immediate invocation **after** the await:
   ```ts
   if (onSuccess) {
     onSuccess(data as UserData);
   }
   ```

3. The surrounding `await autoEnroll(data.id);` remains unchanged, guaranteeing the enroll request has settled before the callback.

## Why this is safe
- No other code paths or components reference `AUTO_ENROLL_DELAY_MS`.
- The `RegistrationSuccessView` UI (shown immediately on `setSuccess(true)`) is unaffected.
- `handleLoginSuccess` behavior is identical; only the timing of when it receives the user object changes (earlier on slow networks).
- Error handling, validation, form reset, and all other branches stay exactly the same.
- The change is a pure removal of an artificial delay that was never intended to gate the callback.

## Implementation order
1. Edit `src/components/RegisterForm.tsx` (remove constant + replace the setTimeout block).
2. Run `npm run lint` and `npm run typecheck` (or equivalent) to verify.
3. (Optional) Manual smoke test: register with a pending event on throttled network; confirm `onSuccess` fires only after enroll request completes.

This plan eliminates the race condition while touching the absolute minimum surface area.
