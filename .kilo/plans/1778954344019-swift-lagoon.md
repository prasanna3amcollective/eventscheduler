# Plan: Add Owner Field to Responsibilities + Polish Form

## Goal
- Add an `owner` field (string) to the `Responsibility` Prisma model.
- Include "Owner" as a required/optional input in the Own Responsibility form.
- Improve `ResponsibilityForm.tsx` styling, structure, and UX to match the quality and polish of `ActivityForm.tsx` (consistent layout, state handling, error/success banners, responsive grid, icons).

## Background / Current State
- Responsibility model (prisma/schema.prisma:31) mirrors Activity but lacks participants/owner.
- Current `ResponsibilityForm.tsx` is a minimal implementation using flat state; uses `activity-form` CSS but lacks the advanced patterns (formData object, overlap warnings, refined handlers) from ActivityForm.
- API (`src/app/api/responsibilities/route.ts`) and validation schema do not yet handle `owner`.
- No existing `owner` references anywhere in codebase.
- "Own Responsibility" button and modal wiring already complete from prior plan.

## Proposed Changes

### 1. Prisma Schema Update
- Add `owner String?` (nullable string, default null) to `Responsibility` model in `prisma/schema.prisma`.
- Keep system columns intact.
- After change: run `npx prisma migrate dev --name add-responsibility-owner` (or `prisma db push` for dev).

### 2. Validation Schema + API Route
- Extend `responsibilitySchema` in `src/app/api/responsibilities/route.ts`:
  - Add `owner: z.string().optional().nullable()`
- Update POST handler:
  - Destructure `owner` from parsedData.
  - Include `owner: owner || null` in the `data` object passed to `withAuth` create.
- (Optional future) Add GET if listing responsibilities is needed, but out of scope for now.

### 3. Frontend Types & Form Component (ResponsibilityForm.tsx)
- Update `ResponsibilityData` interface: add `owner?: string`.
- Refactor form to match ActivityForm quality:
  - Use single `formData` state object (like ActivityForm:270) containing: name, owner, startDateTime, endDateTime, duration, isRecurring, recurrenceFreq, recurrenceDays, category.
  - Implement helper functions: `updateEndFromDuration`, `updateDurationFromTimes`, `buildRrule` (copy/adapt from ActivityForm).
  - Add Owner field as prominent text input in form-grid (icon + label "Owner", placeholder e.g. "Your Name").
  - Keep recurrence UI, date pickers, category select.
  - Reuse existing error/success banners, form-actions.
  - Ensure `handleSubmit` payload includes `owner`.
  - Preserve `onResponsibilityCreated`, `initialData`, `onCancel` props.
- Styling: Ensure all fields use identical classes (`form-field`, `full-width`, `recurring-toggle`, day-btns, etc.) and Icons from `@/components/Icons`.

### 4. Integration Points (No Major Changes Needed)
- `src/app/page.tsx`: Update any local `selectedResponsibility` default object to include `owner: ''`.
- `src/components/CalendarView.tsx`: No changes (button already wired).
- No DB relations added for Owner yet (keep simple string; future enhancement could link to User).

### 5. Styling & UX Polish Goals
- Form should feel identical to Create Activity: clean grid, consistent spacing, icon labels, responsive.
- Support same recurring options (Weekly/Monthly + day toggles).
- Owner field placed right after Name for prominence.
- Handle empty owner gracefully (optional field).

## Files to Modify/Create
- `prisma/schema.prisma` (add field)
- `src/app/api/responsibilities/route.ts` (schema + handler)
- `src/components/ResponsibilityForm.tsx` (major refactor for polish + owner field)
- `src/app/page.tsx` (minor default value update)

## Implementation Order
1. Schema change + migration.
2. Update API + validation.
3. Refactor + enhance ResponsibilityForm.tsx (add owner, match ActivityForm patterns).
4. Minor page.tsx update.
5. Manual test: open modal, fill owner + other fields, submit, verify DB record.

## Open Questions / Tradeoffs (for user input if needed)
- Should `owner` be **required** or optional (nullable)?
- Should Owner use a free-text input or a UserTypeahead dropdown (like Leader in ActivityForm)?
- Add overlap checking for responsibilities (like activities) or keep simple for v1?
- Future: Add `owner` relation to User model + Participant-like table for responsibilities?

This plan keeps changes minimal while delivering the requested field + form quality improvement. Ready for implementation.
